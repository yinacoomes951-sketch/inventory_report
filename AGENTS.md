# Codex 项目协作规范

你是本项目的代码协作助手。你的目标不是快速大范围生成代码，而是在最小修改范围内完成任务，并保证每次修改可评审、可回滚、可追溯。

## 一、默认工作模式

每次接到任务后，必须按以下流程执行：

1. 先分析，不要立即修改代码。
2. 找出与任务最相关的文件、函数、组件或接口。
3. 说明当前逻辑。
4. 给出最小修改方案。
5. 明确预计修改哪些文件。
6. 明确哪些文件不需要修改。
7. 等待我确认后，才能开始修改代码。

如果我明确说“直接修改”或“可以执行”，才允许进入代码修改阶段。

## 二、最小修改原则

1. 每次只处理当前任务。
2. 只修改完成任务必须修改的代码。
3. 不要顺手重构。
4. 不要顺手优化。
5. 不要调整无关命名。
6. 不要大面积格式化。
7. 不要修改任务无关文件。
8. 不要新增依赖，除非任务明确要求。
9. 不要修改 lock 文件，除非任务明确涉及依赖变更。
10. 如果发现必须扩大修改范围，先停止并说明原因，等待确认。

## 三、中间产物规则

1. 不要生成无关中间产物。
2. 不要在源码目录中生成临时文件、草稿文件、备份文件。
3. 如必须生成临时文件，只能放在 `.ai_tmp/` 目录。
4. `.ai_tmp/` 下的内容不得作为最终提交内容。
5. 不要生成 `.bak`、`.backup`、`.orig`、`.tmp` 文件。
6. 不要执行会产生大量无关文件的命令，除非我明确要求。

## 四、命令执行限制

未经明确确认，不要执行以下命令：

- npm install
- pnpm install
- yarn install
- npm run format
- pnpm format
- yarn format
- prettier --write .
- eslint --fix .
- git add .
- git commit
- git push
- git clean -fd
- rm -rf

如果需要运行测试、lint、build，优先运行最小范围命令，不要默认运行全项目命令。

## 五、修改前必须输出

在修改代码前，必须输出：

- 问题理解
- 相关文件
- 当前逻辑
- 最小修改方案
- 预计修改文件
- 不修改范围
- 风险点
- 验证方式

未经确认，不要修改代码。

## 六、修改后必须输出

修改完成后，必须执行或说明 `git status --short` 的结果，并按以下格式输出：

### 实际修改文件

列出本次实际修改的文件。

### 修改说明

说明每个文件为什么修改，修改了什么。

### 变更分类

把 Git 变更分为：

1. 必须保留的业务代码变更
2. 可删除的中间产物
3. 疑似无关变更
4. 建议加入 `.gitignore` 的路径

### 影响范围

说明是否影响：

- 接口参数
- 数据结构
- 配置文件
- 依赖
- 其他页面或模块

### 验证结果

说明：

- 已运行的命令
- 已通过的验证
- 未验证的风险
- 建议人工评审点

## 七、Git 规则

1. 不要自动提交。
2. 不要自动推送。
3. 不要使用 `git add .`。
4. 不要擅自删除未跟踪文件。
5. 修改完成后只说明建议提交哪些文件，由我人工评审后决定。
6. 每次修改应尽量保持 diff 小而清晰。

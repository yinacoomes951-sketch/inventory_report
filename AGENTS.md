# AGENTS.md

# Codex 项目协作规范

你是本项目的代码协作助手。你的目标不是快速、大范围生成代码，而是在**最小修改范围内完成任务**，并保证每次修改都**可评审、可回滚、可追溯**。

本项目优先采用以下工作方式：

* 先分析，再修改。
* 小步修改，禁止大改。
* 所有变更必须能通过 Git diff 清晰评审。
* 测试结果必须有证据，禁止口头声称“已测试通过”。
* 不允许制造大量中间产物污染 Git。
* 不允许自动提交、自动推送。
* 不允许无关重构、无关格式化、无关依赖变更。

---

## 1. 默认工作模式

每次接到任务后，默认必须按以下流程执行：

1. 先阅读相关代码。
2. 先分析问题，不要立即修改代码。
3. 找出与任务最相关的文件、函数、组件、接口或配置。
4. 说明当前逻辑。
5. 给出最小修改方案。
6. 明确预计修改哪些文件。
7. 明确哪些文件不需要修改。
8. 明确风险点和验证方式。
9. 等待确认后，再开始修改代码。

除非用户明确说：

* “可以直接改”
* “直接执行”
* “按最小范围修改”
* “不用等我确认”

否则不要直接修改代码。

```md
- 不改接口结构
- 不改数据库结构
- 不新增依赖

### 7. 风险点

### 8. 验证方式

```

未经确认，不要进入代码修改阶段。

## 3. 最小修改原则

每次修改必须遵守以下原则：

1. 每次只处理当前任务。
2. 只修改完成任务必须修改的代码。
3. 不要顺手重构。
4. 不要顺手优化。
5. 不要调整无关命名。
6. 不要大面积格式化。
7. 不要修改任务无关文件。
8. 不要删除看似无用但未确认的代码。
9. 不要新增依赖，除非任务明确要求。
10. 不要修改 lock 文件，除非任务明确涉及依赖变更。
11. 不要修改配置文件，除非任务明确要求。
12. 不要改动接口入参、出参，除非任务明确要求。
13. 不要改动数据库结构，除非任务明确要求。
14. 如果发现必须扩大修改范围，先停止并说明原因，等待确认。

## 4. 禁止行为

除非用户明确要求，否则禁止以下行为：

```bash
git add .
git commit
git push
git clean -fd
rm -rf
npm install
pnpm install
yarn install
npm run format
pnpm format
yarn format
prettier --write .
eslint --fix .
```

也禁止：

* 全项目格式化。
* 全项目重构。
* 自动修复整个项目 lint。
* 自动升级依赖。
* 自动修改 package-lock.json、pnpm-lock.yaml、yarn.lock。
* 自动生成大量临时文件。
* 自动删除未跟踪文件。
* 自动提交代码。
* 自动推送远程分支。

---

## 5. 中间产物规则

为了避免 Codex 生成中间产物导致 Git 变更文件过多，必须遵守：

1. 不要生成无关中间文件。
2. 不要在源码目录中生成临时文件、草稿文件、备份文件。
3. 如必须生成临时文件，只能放在 `.ai_tmp/` 目录。
4. `.ai_tmp/` 下的内容不得作为最终提交内容。
5. 不要生成以下类型文件：

```text
*.bak
*.backup
*.orig
*.tmp
*.temp
```

6. 不要生成分析草稿、迁移草稿、旧文件副本到源码目录。
7. 如果需要记录分析过程，只在回复中说明，不要写入项目文件。
8. 如果确实需要生成临时文件，完成后必须在变更报告中说明。

建议项目 `.gitignore` 包含：

```gitignore
# AI / Codex temporary files
.ai_tmp/
.codex_tmp/
scratch/
tmp/
*.bak
*.backup
*.orig
*.tmp
*.temp
```

## 6. 命令执行规则

执行命令前必须判断是否会造成大量文件变更。

### 6.1 可以优先执行的安全命令

以下命令通常可以用于查看状态和差异：

```bash
git status --short
git diff
git diff --stat
```

### 6.2 谨慎执行的命令

以下命令只有在任务需要时才执行：

```bash
npm run test
pnpm test
yarn test
npm run lint
pnpm lint
yarn lint
npm run build
pnpm build
yarn build
```

执行前要优先选择最小范围命令。例如只测试相关模块，不默认跑全项目。

### 6.3 禁止默认执行的命令

未经明确确认，不要执行：

```bash
npm install
pnpm install
yarn install
npm run format
pnpm format
yarn format
prettier --write .
eslint --fix .
git add .
git commit
git push
git clean -fd
rm -rf
```

## 8. 测试结果证据规则

禁止只说：

* “测试通过”
* “功能正常”
* “已验证”
* “应该没问题”
* “没有问题”

如果声称已经验证，必须提供证据。

### 8.1 验证报告格式

每次修改后必须输出：

````md
## 验证报告

### 1. 实际执行命令

```bash
填写真实执行过的命令
````

### 2. 命令结果摘要

* 是否成功：
* 是否有报错：
* 关键输出摘要：

### 3. 验证场景

列出已验证的场景，例如：

* 正常输入
* 空值
* 异常值
* 边界条件
* 原有功能回归

### 4. 未验证内容

说明没有验证的内容和原因。

### 5. 结论类型

只能选择以下四类之一：

* 已实际运行并通过
* 已实际运行但失败
* 未运行，仅代码逻辑推断
* 需要人工验证

````

### 8.2 禁止伪造验证结果

1. 不允许声称运行了没有实际运行的命令。
2. 不允许声称测试通过但不提供命令和输出摘要。
3. 不允许把代码推断结果写成测试结果。
4. 如果命令无法运行，必须说明失败原因。
5. 如果因为环境缺失无法验证，必须明确说明。
6. 如果没有实际运行测试，必须写：

```text
未实际运行测试，仅基于代码逻辑推断。
````

## 9. 代码质量自检规则

每次修改后，必须进行代码质量自检。

自检不能代替人工评审，但必须帮助人工评审发现风险。

### 9.1 修改范围检查

* 是否只修改了任务相关文件
* 是否没有无关格式化
* 是否没有顺手重构
* 是否没有新增无关依赖
* 是否没有修改 lock 文件
* 是否没有修改无关配置

### 9.2 可维护性检查

* 逻辑是否清晰
* 命名是否符合项目习惯
* 是否避免重复代码
* 是否没有写死业务值
* 是否没有引入复杂难懂的临时方案
* 是否与项目现有风格一致

### 9.3 稳定性检查

* 是否处理空值
* 是否处理异常返回
* 是否处理边界条件
* 是否没有破坏原有逻辑
* 是否考虑兼容旧数据
* 是否考虑接口返回为空的情况

### 9.4 影响范围检查

* 是否影响接口参数
* 是否影响接口返回
* 是否影响数据库结构
* 是否影响配置
* 是否影响其他页面
* 是否影响其他模块

### 9.5 安全性检查

* 是否没有泄露密钥
* 是否没有写入 token
* 是否没有写入账号密码
* 是否没有写入本地绝对路径
* 是否没有引入危险命令
* 是否没有扩大权限范围

## 10. 代码审查规则

如果用户要求审查代码、审查 diff 或执行 review，只做审查，不要修改文件。

审查时重点检查：

1. 是否实现了本次需求。
2. 是否存在逻辑错误。
3. 是否存在边界条件遗漏。
4. 是否存在空值风险。
5. 是否存在异常处理遗漏。
6. 是否存在无关改动。
7. 是否存在过度设计。
8. 是否存在重复代码。
9. 是否存在性能问题。
10. 是否存在安全风险。
11. 是否存在测试不足。
12. 是否存在 AI 幻觉式验证结论。

审查输出格式：

```md
## 代码审查结果

### 1. 高风险问题
列出必须修复的问题。

### 2. 中风险问题
列出建议修复的问题。

### 3. 低风险建议
列出可选优化建议。

### 4. 无关变更
列出疑似无关文件或 diff。

### 5. 测试与验证问题
指出测试是否缺少证据。

### 6. 结论
- 通过
- 需要修改后再评审
- 不建议合并
```

不要只输出“LGTM”或“看起来没问题”。

## 11. Git 使用规则

1. 不要自动提交。
2. 不要自动推送。
3. 不要使用 `git add .`。
4. 不要擅自删除未跟踪文件。
5. 修改完成后只说明建议提交哪些文件。
6. 由用户人工评审后决定是否提交。
7. 每次修改应尽量保持 diff 小而清晰。

如果需要建议提交命令，只能给出类似：

```bash
git add path/to/file1
git add path/to/file2
git commit -m "fix: 修复xxx问题"
```

不要建议：

```bash
git add .
```

## 13. 需求不明确时的处理

如果任务涉及关键口径、字段含义、接口含义或业务规则不明确，必须先提问确认。

例如以下情况必须先确认：

* 字段含义不明确
* 接口返回结构不明确
* 业务规则存在多种解释
* 改动可能影响多个模块
* 需要新增依赖
* 需要修改数据库结构
* 需要修改接口协议
* 需要做较大范围重构

如果只是小的技术实现细节，可以基于最小影响原则给出建议，但不要直接扩大范围。

## 14. 前端项目特别规则

如果本项目是前端项目，必须额外注意：

1. 不要无关调整 UI 样式。
2. 不要无关修改组件结构。
3. 不要无关修改路由。
4. 不要无关修改全局状态。
5. 不要无关修改接口封装。
6. 修改筛选条件时，要说明对接口请求参数的影响。
7. 修改表格展示时，要说明对空数据、loading、分页、排序、筛选的影响。
8. 修改表单时，要说明对校验、默认值、提交参数的影响。
9. 修改权限逻辑时，要说明对菜单、按钮、路由访问的影响。

前端验证场景至少考虑：

* 正常数据
* 空数据
* 接口异常
* loading 状态
* 筛选条件为空
* 筛选条件有值
* 分页/排序是否受影响

后端验证场景至少考虑：

* 空参数
* 非法参数
* 权限不足
* 数据不存在
* 重复请求

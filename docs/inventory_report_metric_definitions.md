# 诊断报告统计口径 Skill

## 1. 使用目标

使用本 Skill 回答 HTML 库存诊断报告和 `diagnosis JSON` 中各项数据的统计口径、字段来源、计算规则和使用边界。

适用问题示例：

- “诊断报告里的 SPU 数是怎么来的？”
- “整体库存口径包含哪些库存？”
- “预测日销为什么是 SPU 维度？”
- “备货不足 SPU 和发货不足 SPU 是不是互斥？”
- “综合影响分是怎么排序的？”

本 Skill 只覆盖当前诊断报告展示字段、`diagnosis JSON` 字段，以及已补充的 BI 库存查询报表同源上游字段口径。不覆盖源表全部字段。

## 2. 回答流程

回答统计口径问题时按以下顺序处理：

1. 先判断用户问的是哪个字段、页面指标或 JSON 字段。
2. 在第 4 节快速索引中定位字段类型。
3. 到第 5-15 节查对应口径。
4. 回答时优先用业务语言说明，再补充技术口径或代码链路。
5. 如果本 Skill 没有记录该字段或规则，不要推测，不要补造公式。

### 2.1 未查询到口径时的标准回答

当本 Skill 没有记录某个统计口径时，只能按以下方式回答：

```text
未查询到该统计口径。当前 Skill 中没有记录该字段的上游计算规则或业务定义，不能基于现有信息推断。建议补充该字段的来源、计算公式、统计粒度和适用范围后再回答。
```

如果能确认“报告代码只是直接读取字段”，但不知道上游怎么计算，可以回答：

```text
当前报告代码只直接读取该字段值，没有在报告层重新计算；但本 Skill 未记录该字段在 BI 库存查询报表同源底层数据中的上游统计口径，因此不能继续解释其来源公式。
```

### 2.2 禁止回答方式

- 不要说“应该是”“通常是”“可能是”来补齐缺失口径。
- 不要根据字段名猜公式。
- 不要把 SKU 层字段直接说成 SPU 层口径。
- 不要把源表上游公式说成报告代码重新计算。
- 不要把建议动作说成系统自动决策。

## 3. 全局回答规则

1. 只能基于本 Skill、当前 `diagnosis JSON` 和已提供的报告内容回答。
2. 不重新计算用户没有提供的明细数据，不补造 SKU、SPU、数量、比例或原因。
3. 区分三类字段：
   - 源表字段：业务上可理解为与 BI 中“库存查询报表”同源的原始字段；技术追溯时对应后端读取的库存源表字段。
   - 报告聚合字段：后端按报告范围、SPU 或 SKU 聚合得到的字段。
   - 派生判断字段：由阈值、排序或诊断规则生成的状态、问题、建议。
4. 说明业务原因时必须谨慎。库存表能证明“库存、销量、库龄、覆盖天数等事实”，不能直接证明停售、链接异常、新品冷启动、活动影响等业务原因。
5. 覆盖天数为空时说明“预测日销分母无效”，不要解释成 `0天`。
6. 问题 SPU 数之间不是全部互斥，不能简单相加后与 SPU 总数比较。
7. 采购、发货、清仓等最终动作必须表述为“建议复核/建议排查”。

## 5. 数据来源链路

### 5.1 业务来源

报告底层数据与 BI 中“库存查询报表”的数据来源一致。业务人员理解时，可以先按 BI 库存查询报表中的库存、销量、预测日销、库龄等字段口径理解。

### 5.2 技术链路

```text
BI库存查询报表同源底层数据
-> 最新 insert_time 批次
-> 角色范围过滤
-> _report_inventory_cte()
-> _scope_metrics() / _spu_health() / _warning_distribution() / _top_risk_skus()
-> InventoryDiagnosisEngine.build()
-> report_renderer.render_html()
```

### 5.3 关键代码位置

| 代码位置 | 作用 |
| --- | --- |
| `backend/src/ai_inventory_backend/repository.py::_latest_batch()` | 选择最新 `insert_time` 批次 |
| `backend/src/ai_inventory_backend/repository.py::_build_scopes()` | 生成运营个人、战队/部门、运营总监三个报告范围 |
| `backend/src/ai_inventory_backend/repository.py::_report_inventory_cte()` | 统一角色过滤、预测日销归零、SPU 纳入规则 |
| `backend/src/ai_inventory_backend/repository.py::_scope_metrics()` | 生成报告总览聚合指标 |
| `backend/src/ai_inventory_backend/repository.py::_spu_health()` | 生成 SPU 健康状态、问题分布、核心影响 SPU |
| `backend/src/ai_inventory_backend/repository.py::_warning_distribution()` | 生成源表 `备货预警` 标签分布 |
| `backend/src/ai_inventory_backend/repository.py::_top_risk_skus()` | 生成 SKU 证据列表 |
| `backend/src/ai_inventory_backend/diagnosis.py::InventoryDiagnosisEngine.build()` | 组装 `diagnosis JSON` |
| `backend/src/ai_inventory_backend/report_renderer.py::InventoryReportRenderer.render_html()` | 将 `diagnosis JSON` 渲染成 HTML 报告 |

## 6. 数据范围

### 6.1 数据批次

技术实现上，后端读取库存源表最新 `insert_time` 批次的数据。不同报告对象使用同一个最新批次，不混用历史批次。

### 6.2 角色范围

| 报告对象 | 过滤口径 |
| --- | --- |
| 运营个人 | 按源表 `归属` 过滤 |
| 战队/部门 | 按源表 `部门名称` 过滤 |
| 运营总监 | 不加 `归属` 或 `部门名称` 过滤，仍受最新批次限制 |

### 6.3 SPU 纳入规则

数据先按角色范围过滤，再按 SPU 汇总判断是否纳入报告。

- SPU 汇总 `预测日销` 严格小于 `0.5` 时，报告按 `0` 处理。
- SPU 汇总 `预测日销` 等于 `0.5` 时，保留原值。
- 仅当 SPU 的 `总库存` 汇总值和处理后的预测日销同时为 `0` 时，该 SPU 不纳入报告。
- 库存不为 `0` 但处理后预测日销为 `0` 的 SPU 会保留，用于识别无动销或呆滞风险。

## 7. `diagnosis JSON` 顶层结构

| 字段 | 类型 | 口径 |
| --- | --- | --- |
| `scope` | 对象 | 当前报告范围，包含 `level`、`object_name`、`batch_key` |
| `summary` | 对象 | 报告总览，包括风险等级、健康状态、关键数量、库存和库龄汇总 |
| `problems` | 数组 | 按诊断规则生成的 TOP 问题，不是源表原始问题 |
| `warning_distribution` | 数组 | 源表 `备货预警` 标签分布，按 SKU 行数统计 |
| `spu_health` | 对象 | SPU 健康状态分布、问题分布、核心影响 SPU |
| `charts` | 对象 | HTML 报告图表使用的数据 |
| `top_skus` | 数组 | SKU 层证据列表，默认最多取 20 条风险证据 |
| `action_list` | 对象 | 从 `problems` 派生的行动清单 |

## 8. 总体判断字段

### 8.1 `summary.risk_level`

`risk_level` 是报告风险等级，来自 SKU 层源表 `备货预警` 统计：

| 返回值 | 判断口径 |
| --- | --- |
| `high` | `缺货预警` SKU 数量大于等于 10，或缺货 SKU 占全部 SKU 比例大于等于 8% |
| `medium` | 存在 `缺货预警` SKU，或 `限制备货` SKU 数量大于等于 10 |
| `low` | 未命中以上规则 |

该字段用于报告风险提示，不等同于财务风险等级或最终业务优先级。

### 8.2 `summary.health_status`

`health_status` 是报告整体健康状态，由 SPU 健康状态分布汇总：

| 返回值 | 页面含义 | 判断口径 |
| --- | --- | --- |
| `stagnant` | 呆滞风险突出 | 呆滞风险 SPU 数大于等于 3 |
| `unhealthy` | 不健康 | 存在至少 1 个不健康 SPU |
| `local_warning` | 局部预警 | 不存在不健康 SPU，但存在至少 1 个局部预警 SPU |
| `healthy` | 整体健康 | 未命中以上规则 |

整体状态是快速提示，不代表所有 SPU 都处于同一状态。

## 9. 数量类字段

### 9.1 `summary.key_counts`

| 字段 | 页面名称 | 统计口径 |
| --- | --- | --- |
| `spu_count` | SPU 数 | 报告范围内纳入诊断的不同 SPU 数量 |
| `restock_shortage_spu_count` | 备货不足 SPU | 整体可售天数 `< 90` 的 SPU 数 |
| `restock_excess_spu_count` | 备货过量 SPU | 整体可售天数 `> 120` 的 SPU 数 |
| `shipment_shortage_spu_count` | 发货不足 SPU | 在途+可售天数 `< 60` 的 SPU 数 |
| `shipment_excess_spu_count` | 发货过量 SPU | 在途+可售天数 `> 80` 的 SPU 数 |
| `no_movement_spu_count` | 无动销 SPU | 总库存大于 0，且近30天销量不大于 0，且处理后预测日销为 0 的 SPU 数 |

同一个 SPU 可以同时进入多个问题分类，因此这些问题 SPU 数不能直接相加等于 `spu_count`。

## 10. 库存、销量、库龄字段

### 10.1 `summary.totals`

| 字段 | 页面/业务名称 | 来源与计算口径 |
| --- | --- | --- |
| `total_inventory` | 总库存 | `Σ 总库存`；源表上游口径为 `国内合计 + 国外合计` |
| `stocking_inventory_qty` | 整体库存口径 | 当前等于 `total_inventory`，即 `Σ 总库存` |
| `overseas_ready_qty` | 海外在途+可售 | `Σ 国外合计`；源表上游口径见第 10.2 节 |
| `domestic_total_qty` | 国内库存 | `Σ 国内总数量`；源表上游口径见第 10.2 节 |
| `fba_available` | FBA 可售 | `Σ fba可售` |
| `awd_available` | AWD 可用 | `Σ awd可用` |
| `purchase_in_transit_qty` | 采购在途 | `Σ 采购在途` |
| `purchase_plan_qty` | 采购计划 | `Σ 采购计划` |
| `suggested_restock_qty` | 建议备货量 | `Σ 建议备货量`；源表上游口径见第 12.2 节 |
| `avg_sellable_days` | 平均可售天数 | 源表 `可售天数` 去掉 0 后求平均；源表上游口径见第 12.3 节 |
| `sales_30d` | 近30天销量 | `Σ 最近30天总销量`；当该值为 0 时，业务上等价于近期日销为 0 |
| `domestic_aged_90_qty` | 国内90天以上库龄 | 国内 90 天及以上库龄字段合计 |
| `overseas_aged_90_qty` | 海外90天以上库龄 | 海外 3 个月及以上库龄字段合计 |
| `aged_90_qty` | 90天以上库龄 | `domestic_aged_90_qty + overseas_aged_90_qty` |
| `domestic_aged_qty` | 国内180天以上库龄 | 国内 180 天及以上库龄字段合计 |
| `overseas_aged_qty` | 海外6个月以上库龄 | 海外 6 个月及以上库龄字段合计 |
| `aged_12m_qty` | 12个月以上库龄 | `国内库龄_365以上 + 12个月以上库龄` |

### 10.2 总库存组成

```text
总库存 = 国内合计 + 国外合计
国内合计 = 采购计划 + 采购在途 + 自备货仓 + 个人仓 + VMI仓
国外合计 = 实际在途 + FBA入库中 + FBA可售 + FBA调仓中 + FBA待调仓
         + AWD实际在途 + AWD可用 + AWD待发货 + AWD标发在途数量
```

报告层使用时仍以源表已经生成的 `总库存`、`国内总数量`、`国外合计` 字段为准，不在报告代码中重新展开上述组成项计算。

### 10.3 库龄口径

```text
国内90天以上库龄 = 国内库龄_90_180 + 国内库龄_180_270 + 国内库龄_270_330 + 国内库龄_330_365 + 国内库龄_365以上
国内180天以上库龄 = 国内库龄_180_270 + 国内库龄_270_330 + 国内库龄_330_365 + 国内库龄_365以上
海外90天以上库龄 = 3_6个月库龄 + 6_9个月库龄 + 9_11个月库龄 + 11_12个月库龄 + 12个月以上库龄
海外6个月以上库龄 = 6_9个月库龄 + 9_11个月库龄 + 11_12个月库龄 + 12个月以上库龄
12个月以上库龄 = 国内库龄_365以上 + 12个月以上库龄
```

海外数据按月划分库龄区间，因此报告将 `3_6个月库龄` 及以上作为海外 90 天以上库存。

## 11. 预测日销与覆盖天数

### 11.1 预测日销

报告统一使用源表 `预测日销` 作为覆盖天数分母。报告代码不重新计算 SKU 预测日销，只直接使用源表已有的 `预测日销` 字段。

源表上游预测日销口径：

```text
SPU预测日销 = 该 SPU 下属所有 SKU预测日销之和
SKU预测日销 = SKU近期日销 * 季节系数倍数
季节系数倍数 = 后六个月平均季节系数 / 当前月季节系数
```

季节系数使用欧洲或美洲口径，取决于该 SKU 所属负责人的区域。

报告层处理规则：

```text
SPU报告预测日销 = Σ 同一 SPU 下 SKU 源表预测日销
SPU报告预测日销 < 0.5 -> 0
SPU报告预测日销 >= 0.5 -> 保留汇总后的预测日销
```

总览覆盖天数使用报告范围内处理后的预测日销合计；SPU 表覆盖天数使用单个 SPU 的处理后预测日销合计。

### 11.2 整体可售天数

```text
整体可售天数 = 整体库存口径 ÷ 处理后预测日销
健康参考区间 = 90 至 120 天
```

- `< 90`：备货不足。
- `> 120`：备货过量。
- `90` 和 `120` 本身不触发不足或过量。

整体可售天数用于采购/备货判断，不等同于海外发货可支撑天数。

### 11.3 在途+可售天数

```text
在途+可售天数 = 海外在途+可售 ÷ 处理后预测日销
健康参考区间 = 60 至 80 天
```

- `< 60`：发货不足。
- `> 80`：发货过量。
- `60` 和 `80` 本身不触发不足或过量。

该指标只用于判断海外覆盖和发货节奏，不包含国内库存、采购在途和采购计划。

### 11.4 覆盖天数为空

当处理后预测日销为 `0` 时，覆盖天数返回空值，报告显示为 `-`。回答时应解释为“没有有效日销分母”，不要解释为 `0天`。

## 12. BI 同源上游字段口径

以下字段由 BI 库存查询报表同源底层数据提前计算，报告代码直接读取字段值，不在报告层重新计算。

### 12.1 备货预警

`备货预警` 是 SKU 层标签字段，按 SKU 的 `可售天数` 和 `预测日销` 判断：

| 标签 | 判断口径 |
| --- | --- |
| `限制备货` | `可售天数 > 120` |
| `正常` | `90 < 可售天数 <= 120` |
| `缺货预警` | `可售天数 < 90` |
| `无销量` | `预测日销 = 0` |

回答时应说明该字段是源表上游标签。诊断报告会把它作为 SKU 证据和风险等级参考，但报告主判断仍以 SPU 汇总后的覆盖天数、库龄和动销规则为主。

### 12.2 建议备货量

```text
建议备货量 = (建议可售天数 - 可售天数) * 预测日销
```

该字段是 SKU 层建议量。报告总览中的 `suggested_restock_qty` 是报告范围内 `Σ 建议备货量`。

### 12.3 可售天数

```text
可售天数 = 总库存 / 预测日销
```

源表 `可售天数` 是 SKU 层字段。报告诊断自己的整体可售天数会在 SPU 或报告范围内重新用汇总后的库存和预测日销计算，因此不要把 SKU 源表 `可售天数` 直接当作 SPU 诊断口径。

### 12.4 建议可售天数

```text
备货建议可售天数 = 120天
发货建议可售天数 = 80天
```

`建议可售天数` 是常量口径。

### 12.5 近期日销

```text
近期日销 = 近7天平均日销 * 10%
         + 近15天平均日销 * 20%
         + 近21天平均日销 * 30%
         + 近30天平均日销 * 40%
```

近期日销是预测日销的上游输入之一。业务上，当 `最近30天总销量 = 0` 时，可理解为近期日销为 0。

## 13. 问题 SPU 口径

| 问题类型 | `spu_health.problem_distribution` 字段 | 判断口径 | 业务含义 |
| --- | --- | --- | --- |
| 备货不足 SPU | `restock_shortage_spu` | 整体可售天数 `< 90` | 全部库存覆盖不足，需复核采购在途、采购计划和预测日销 |
| 备货过量 SPU | `restock_excess_spu` | 整体可售天数 `> 120` | 全部库存覆盖偏高，需控制新增备货 |
| 发货不足 SPU | `shipment_shortage_spu` | 在途+可售天数 `< 60` | 海外供给不足，优先考虑发货或调拨 |
| 发货过量 SPU | `shipment_excess_spu` | 在途+可售天数 `> 80` | 海外供给偏高，优先控制继续发货 |
| 无动销 SPU | `no_movement_spu` | 总库存大于 0，近30天销量不大于 0，处理后预测日销为 0 | 有库存但没有有效动销，需要人工排查原因；近30天销量为 0 时，业务上等价于近期日销为 0 |
| 健康 SPU | `healthy_spu` | 未触发以上问题分类 | 当前未触发主要问题规则 |

`healthy_spu` 是问题分布中的剩余项；其他问题分类之间不互斥。

## 14. SPU 健康、影响分与排序

### 14.1 SPU 健康状态

每个 SPU 只有一个主健康状态。判断按以下顺序执行，先命中的状态生效。

| 优先级 | `health_status` | 页面含义 | 判断口径 |
| --- | --- | --- | --- |
| 1 | `stagnant` | 呆滞风险 | 处理后预测日销 `<= 0` 且总库存 `> 0` |
| 2 | `unhealthy` | 不健康 | 整体可售天数 `< 90`，或在途+可售天数 `< 60` |
| 3 | `local_warning` | 局部预警 | 整体可售天数 `> 120`，或在途+可售天数 `> 80`，或存在 90 天以上库龄 |
| 4 | `healthy` | 健康 | 未命中以上规则 |

### 14.2 综合影响分

综合影响分用于核心影响 SPU 排序，分数越高表示越值得优先排查。它不是风险概率、健康百分比或金额。

```text
综合影响分
= 整体可售天数偏离得分
+ 在途+可售天数偏离得分
+ 呆滞附加分
+ 90天以上库龄附加分
```

| 风险项 | 计分方式 |
| --- | --- |
| 整体可售天数低于 90 天 | `(90 - 实际天数) * 8` |
| 整体可售天数高于 120 天 | `(实际天数 - 120) * 3` |
| 在途+可售天数低于 60 天 | `(60 - 实际天数) * 7` |
| 在途+可售天数高于 80 天 | `(实际天数 - 80) * 3` |
| 处理后预测日销为 0 且总库存大于 0 | 加 160 分 |
| 90 天以上库龄 | `90天以上库龄 / 100`，最高加 80 分 |

### 14.3 核心影响 SPU 排序

核心影响 SPU 来自 `spu_health.problem_top_spus`，按问题组分别取最多 8 个 SPU。一个 SPU 可以出现在多个问题组。

| 问题组 | 排序口径 |
| --- | --- |
| `restock_shortage` | 整体可售天数越低越靠前；相同或接近时按预测日销、综合影响分降序 |
| `restock_excess` | 整体可售天数越高越靠前；再按 90 天以上库龄、综合影响分降序 |
| `shipment_shortage` | 在途+可售天数越低越靠前；再按预测日销、综合影响分降序 |
| `shipment_excess` | 在途+可售天数越高越靠前；再按 90 天以上库龄、综合影响分降序 |
| `stagnant` | 12个月以上库龄、90天以上库龄、总库存、综合影响分依次降序 |

## 15. 证据、问题、行动与图表字段

### 15.1 `warning_distribution`

`warning_distribution` 来自源表 `备货预警` 字段，按 SKU 行数分组统计：

| 字段 | 口径 |
| --- | --- |
| `warning` | 源表 `备货预警`，为空时显示 `未标记` |
| `count` | 该预警标签下的 SKU 行数 |
| `suggested_restock_qty` | 该预警标签下 `Σ 建议备货量`；建议备货量上游公式见第 12.2 节 |

该分布是源表标签视角，不等同于报告的 SPU 诊断问题分布。报告判断以 SPU 汇总后的覆盖天数、库龄和动销规则为主。

### 15.2 `top_skus`

`top_skus` 是 SKU 层证据列表，不是报告主判断粒度。默认最多取 20 条，排序规则：

1. `备货预警 = 缺货预警`
2. `备货预警 = 限制备货`
3. `备货预警 = 无销量`
4. 其他标签
5. 同组内按 `建议备货量` 降序，再按 `最近30天总销量` 降序

`top_skus[].warning` 会把源表标签转成报告表达：

| 源表 `备货预警` | `top_skus[].warning` |
| --- | --- |
| `缺货预警` | 保供风险 |
| `限制备货` | 库存过量 |
| `无销量` | 无动销 |
| `正常` | 健康 |
| 其他或空值 | 原值或 `未标记` |

`top_skus[].facts` 中的覆盖天数按单个 SKU 的库存字段除以该 SKU 的源表 `预测日销` 计算，仅用于证据参考，不能替代 SPU 层诊断。

### 15.3 `problems`

`problems` 是规则诊断后的问题列表，由聚合指标和 SPU 问题分布生成。常见 `problem_type`：

| `problem_type` | 触发口径 | 说明 |
| --- | --- | --- |
| `stockout` | 存在备货不足 SPU | 关注整体库存水位不足 |
| `restrict_restock` | 存在备货过量 SPU | 关注整体库存水位过高 |
| `shipment` | 存在发货不足或发货过量 SPU | 关注海外在途+可售水位 |
| `no_sales` | 存在无动销 SPU | 关注有库存但没有有效动销 |
| `aged_inventory` | 存在长库龄库存或呆滞 SPU | 关注清仓、移除、停售复核或坏账风险 |
| `spu_health` | 存在不健康 SPU | 提示从 SPU 下钻到 SKU |
| `shipment_allocation` | 未发现高优先级库存问题 | 保持常规监控 |

`facts` 必须来自诊断字段，不得加入外部猜测。`inference` 是基于规则的解释，`action` 是建议动作，不是自动执行结果。

### 15.4 `action_list`

| 字段 | 来源 |
| --- | --- |
| `today` | `problems` 中优先级为 `P0` 的行动建议 |
| `this_week` | `problems` 中优先级为 `P1` 或 `P2` 的行动建议 |
| `human_check` | `problems` 中 `needs_human_check = true` 的排查问题 |

如果没有对应问题，系统会返回默认兜底建议。

### 15.5 `charts`

| 字段 | 图表 | 来源 |
| --- | --- | --- |
| `spu_health_distribution` | SPU 健康分布 | `spu_health.distribution` |
| `warning_distribution` | 库存问题分布 | `spu_health.problem_distribution`，不是源表 `备货预警` 分布 |
| `coverage` | 覆盖区间 | 总览整体可售天数、在途+可售天数及对应健康区间 |
| `aging` | 库龄分布 | 国内90天以上、海外90天以上、12个月以上库龄 |

## 16. 常见回答边界

- 数据来源对业务人员优先说“与 BI 中库存查询报表同源”，需要技术追溯时再说明后端读取库存源表最新批次。
- “整体库存口径”当前就是源表 `总库存` 合计，不要解释为程序现场把各库存字段相加。
- “海外在途+可售”来自 `国外合计`，不包含国内库存、采购在途、采购计划。
- “预测日销”的季节系数公式是源表上游口径，报告代码只读取源表 `预测日销` 并在 SPU 层汇总。
- “源表可售天数”是 SKU 层字段，报告 SPU 诊断会重新使用汇总后的库存和预测日销计算覆盖天数。
- “备货不足”看整体可售天数，不是只看海外库存。
- “发货不足”看海外在途+可售天数，不直接推出需要采购。
- “无动销”只能说明有库存但没有有效动销，具体原因必须人工排查；最近30天总销量为 0 可理解为近期日销为 0。
- “源表备货预警”是辅助标签，最终报告主判断以 SPU 汇总指标为准。
- “核心影响 SPU 表”是优先排查清单，不等于完整明细清单。
- “综合影响分”只能用于同一版本规则下的相对排序，不能作为跨版本 KPI。

# 库存报告表达 Skill

## 目标

把结构化 diagnosis 转为现代、易扫描、可行动的中文 HTML 报告。报告负责解释，不新增数字或原因。

报告字段名称、统计口径和使用边界详见 `inventory_report_metric_definitions.md`，表达时不得改变该文档定义的字段含义。

## 阅读顺序

1. 当前库存是否健康。
2. 问题集中在哪些 SPU。
3. 分别属于整体库存、海外覆盖、库龄还是无动销。
4. 下一步应发货、复核采购、控制库存、清理，还是先查数据。
5. 需要明细时点击 BI 下钻，不在首屏堆 SKU。

## 报告结构

1. 核心结论：一句话状态、主要矛盾、三个最优先动作。
2. 概念注释：解释健康、局部预警、不健康、呆滞风险。
3. 库存健康总览：SPU 数、问题 SPU 数、库存量和库龄。
4. 图表概览：SPU 健康分布、问题分布、覆盖区间、库龄分布。
5. 按问题展开核心 SPU：
   - 整体库存不足；
   - 整体库存过量；
   - 海外覆盖不足；
   - 海外覆盖过量；
   - 无动销与长库龄。
6. BI 排查建议：提供筛选条件、字段和预期决策。
7. 行动清单：今天、本周、需人工确认。

## 表达约束

- 前半部分只说 SPU，不展示大面积 SKU 指标卡或明细表。
- SKU 仅在极端个案或证据层出现，默认使用 BI 跳转。
- 使用“整体库存覆盖天数”和“海外在途及可售覆盖天数”。
- 不使用“备货天数”“发货天数”“缺货预警”“限制备货”作为报告主概念。
- 同类问题分开成组，不能把所有 SPU 放进一个混合表。
- 每个问题必须说明：为什么重要、事实证据、先查什么、建议动作。
- 图表服务于比较和定位，不为装饰而添加。

## 质量标准

- 用户 30 秒内能知道最先处理的三件事。
- 每个结论都能追溯到结构化字段。
- 不把国内库存不足直接写成采购建议。
- 不把单个 SKU 无销量扩大为整个 SPU 无动销。
- 不确定原因明确写“需人工确认”。
- 报告正文不得泄露无关人员或超出角色数据范围的信息。
